Arcana SSO v2 - Guida Operativa

Questa guida descrive il funzionamento del mu-plugin arcana-sso-v2, i flussi di autenticazione supportati, l'ordine corretto delle chiamate e i parametri necessari per integrare un sito esterno o un sottodominio con il sistema SSO di Arcana Domine.

Il plugin implementa un flusso OAuth2 Authorization Code con PKCE e usa JWT firmati HS256 per l'access token.

1. Obiettivo del sistema

Il mu-plugin arcana-sso-v2 serve a:

  1. autenticare un utente su Arcana Domine come identity provider;
  2. trasferire in modo sicuro l'identita' dell'utente a un sito client;
  3. permettere al sito client di creare la propria sessione locale;
  4. evitare di condividere cookie WordPress tra domini diversi;
  5. evitare di usare gli ID locali dei database client come identita' globale.

Il sistema e' adatto a:

  1. siti campagna custom PHP;
  2. siti WordPress separati;
  3. sottodomini Arcana Domine con database diversi;
  4. applicazioni custom non WordPress.

2. Concetti chiave

2.1 Identity provider

Arcana Domine e' il provider centrale dell'identita'.

L'identita' globale dell'utente e' rappresentata principalmente da:

  1. sub
  2. arcanadomine_id

Nel sistema attuale questi due valori coincidono con l'ID utente WordPress del sito principale Arcana Domine.

2.2 Client

Il sito esterno o sottodominio che vuole autenticare l'utente tramite Arcana Domine viene registrato come client OAuth2.

Ogni client ha:

  1. client_id
  2. client_secret
  3. una o piu' redirect_uri

2.3 Mapping locale degli utenti

Il sito client non deve mai considerare l'ID numerico di Arcana Domine come ID locale del proprio database.

Approccio corretto:

  1. ricevere sub o arcanadomine_id da /userinfo;
  2. cercare nel database locale un mapping verso l'utente locale;
  3. se il mapping non esiste, creare o collegare l'utente locale;
  4. usare da quel momento l'ID locale del sito client.
Questo evita l'errore tipico dei sottodomini legacy in cui lo stesso ID numerico viene riutilizzato su database diversi per utenti diversi.

3. Architettura del flusso

Il sistema usa:

  1. OAuth2 Authorization Code Flow;
  2. PKCE obbligatorio con metodo S256;
  3. Access Token JWT HS256;
  4. Refresh Token persistito lato server;
  5. endpoint userinfo per il recupero del profilo.

4. Flussi supportati

4.1 Flow A - utente gia' loggato su Arcana Domine

  1. l'utente e' gia' autenticato su Arcana Domine;
  2. clicca un link verso il sito campagna;
  3. il sito campagna avvia il flusso OAuth2;
  4. Arcana Domine vede che l'utente e' gia' loggato;
  5. genera un authorization code;
  6. reindirizza il browser al client con code e state;
  7. il client scambia il code per access_token e refresh_token;
  8. il client chiama /userinfo;
  9. il client crea la propria sessione locale.

4.2 Flow B - utente parte dal sito client e non e' loggato su AD

  1. il client reindirizza l'utente a /authorize;
  2. Arcana Domine rileva che l'utente non e' loggato;
  3. AD reindirizza l'utente alla pagina di login WordPress;
  4. dopo il login, AD riprende il flusso;
  5. AD genera authorization code;
  6. AD reindirizza il browser al client con code e state;
  7. il client chiama /token;
  8. il client chiama /userinfo;
  9. il client crea la sessione locale.

5. Endpoints disponibili

Base URL:

https://DOMINIO-AD/wp-json/ad-sso/v2

Endpoint disponibili:

  1. GET /authorize
  2. POST /token
  3. GET /userinfo
  4. POST /revoke

Esiste anche un logout front-channel su dominio Arcana Domine:

  1. GET /?ad_sso_logout=1&client_id=...&return_to=...

6. Ordine corretto delle chiamate

  1. il client genera state, code_verifier, code_challenge;
  2. il browser viene mandato a GET /authorize;
  3. Arcana Domine reindirizza il browser verso la callback del client con code e state;
  4. il client verifica state;
  5. il client fa POST /token;
  6. il client riceve access_token e refresh_token;
  7. il client fa GET /userinfo con Bearer token;
  8. il client crea o aggiorna la sessione locale;
  9. opzionalmente, al logout, il client chiama POST /revoke;
  10. opzionalmente, il client effettua logout front-channel su Arcana Domine.

7. Dettaglio endpoint

7.1 Authorize

Endpoint:

GET /wp-json/ad-sso/v2/authorize

Scopo:

  1. avviare il flusso OAuth2;
  2. autenticare l'utente su AD se necessario;
  3. generare un authorization code monouso;
  4. reindirizzare il browser alla callback del client.

Parametri richiesti

Parametro Obbligatorio Descrizione
response_typesiDeve essere sempre code
client_idsiIdentificativo del client registrato
redirect_urisiDeve corrispondere esattamente a una URI registrata per il client
statesiValore casuale generato dal client per protezione CSRF
code_challengesiPKCE challenge derivata dal code_verifier
code_challenge_methodsiDeve essere sempre S256
scopenoAttualmente openid profile

Esempio richiesta

GET https://arcanadomine.local/wp-json/ad-sso/v2/authorize?response_type=code&client_id=client_123&redirect_uri=https%3A%2F%2Fcampagna.local%2Fcallback.php&state=abc123&code_challenge=XYZ&code_challenge_method=S256&scope=openid%20profile

Risultato atteso

https://campagna.local/callback.php?code=AUTH_CODE&state=abc123

Errori possibili

ErroreSignificato
invalid_requestParametri mancanti o non validi
unsupported_response_typeresponse_type diverso da code
unknown_clientclient_id non registrato
invalid_redirect_uriredirect_uri non ammessa per quel client

7.2 Token

Endpoint:

POST /wp-json/ad-sso/v2/token

Scopo:

  1. scambiare un authorization code con token validi;
  2. rinnovare l'access token tramite refresh token.

Modalita' 1 - grant_type authorization_code

Parametro Obbligatorio Descrizione
grant_typesiDeve essere authorization_code
codesiAuthorization code ricevuto da /authorize
redirect_urisiDeve essere identica a quella usata in /authorize
client_idsiClient registrato
client_secretsiSegreto del client
code_verifiersiValore PKCE originario usato per generare il challenge
 
curl -X POST https://arcanadomine.local/wp-json/ad-sso/v2/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=authorization_code" \
  -d "code=AUTH_CODE" \
  -d "redirect_uri=https://campagna.local/callback.php" \
  -d "client_id=client_123" \
  -d "client_secret=CLIENT_SECRET" \
  -d "code_verifier=ORIGINAL_CODE_VERIFIER"

Risposta di successo:

{
  "access_token": "JWT_TOKEN",
  "token_type": "Bearer",
  "expires_in": 900,
  "refresh_token": "REFRESH_TOKEN",
  "scope": "openid profile"
}

Modalita' 2 - grant_type refresh_token

Parametro Obbligatorio Descrizione
grant_typesiDeve essere refresh_token
refresh_tokensiRefresh token valido
client_idsiClient registrato
client_secretsiSegreto del client
 
curl -X POST https://arcanadomine.local/wp-json/ad-sso/v2/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=refresh_token" \
  -d "refresh_token=REFRESH_TOKEN" \
  -d "client_id=client_123" \
  -d "client_secret=CLIENT_SECRET"

Errori possibili

ErroreSignificato
invalid_requestParametri mancanti
invalid_clientclient_id o client_secret non validi
invalid_grantcode o refresh_token non validi, scaduti o gia' usati
unsupported_grant_typegrant_type non supportato
server_errorerrore interno lato AD

7.3 UserInfo

Endpoint:

GET /wp-json/ad-sso/v2/userinfo

Scopo:

  1. restituire il profilo dell'utente associato all'access token;
  2. permettere al client di creare il mapping locale utente.

Header richiesto

Authorization: Bearer ACCESS_TOKEN

Esempio richiesta

curl https://arcanadomine.local/wp-json/ad-sso/v2/userinfo \
  -H "Authorization: Bearer JWT_TOKEN"

Esempio risposta

{
  "sub": 40,
  "iss": "https://arcanadomine.local",
  "aud": "client_123",
  "arcanadomine_id": 40,
  "username": "nico.moro",
  "nome": "Nico",
  "cognome": "Moro",
  "email": "nico@example.com",
  "tipologia": "pg",
  "codice_fiscale": "ABCDEF00A00A000A",
  "ruoli": ["registrato"],
  "stato": "inattivo"
}

Campi rilevanti per il mapping locale

CampoUso consigliato
subChiave globale primaria dell'identita' AD
arcanadomine_idAlias attuale dell'identita' AD
emailSupporto per lookup o sync dati
usernameSupporto per sincronizzazione profilo

Errori possibili

ErroreSignificato
invalid_tokenToken assente, non valido o scaduto

7.4 Revoke

Endpoint:

POST /wp-json/ad-sso/v2/revoke

Scopo:

  1. revocare un refresh token;
  2. invalidare futuri rinnovi del token sul client.

Parametri richiesti

Parametro Obbligatorio Descrizione
tokensiRefresh token da revocare
client_idsiClient registrato
client_secretsiSegreto del client
 
curl -X POST https://arcanadomine.local/wp-json/ad-sso/v2/revoke \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "token=REFRESH_TOKEN" \
  -d "client_id=client_123" \
  -d "client_secret=CLIENT_SECRET"

Risposta attesa:

  1. 200 OK anche se il token era gia' revocato o inesistente;
  2. comportamento coerente con RFC 7009.

7.5 Logout front-channel su Arcana Domine

Scopo:

  1. cancellare la sessione WordPress sul dominio Arcana Domine;
  2. evitare che l'utente rimanga loggato su AD dopo il logout dal sito client.

Endpoint:

GET https://DOMINIO-AD/?ad_sso_logout=1&client_id=CLIENT_ID&return_to=URL_DI_RITORNO

Parametri:

Parametro Obbligatorio Descrizione
ad_sso_logoutsiDeve valere 1
client_idsiClient registrato
return_tosiURL a cui AD deve tornare dopo il logout

Note:

  1. return_to viene accettato solo se il suo origin appartiene a una redirect_uri registrata per quel client;
  2. questo endpoint esegue wp_logout() e pulisce il cookie WordPress di Arcana Domine;
  3. il browser viene poi reindirizzato al client.

8. PKCE - come generare verifier e challenge

Il client deve generare:

  1. un code_verifier casuale;
  2. un code_challenge ottenuto con SHA256 e base64url.

Pseudocodice:

$codeVerifier = base64url(random_bytes(64));
$codeChallenge = base64url(hash('sha256', $codeVerifier, true));

Regole:

  1. il code_verifier va tenuto solo dal client;
  2. il code_challenge viene passato a /authorize;
  3. il code_verifier viene passato solo a /token.

9. Sessione locale sul client

Il client, dopo aver ottenuto access_token e userinfo, deve creare una propria sessione locale.

Passi consigliati:

  1. cercare l'utente locale tramite sub;
  2. se non esiste, crearlo o associare un account esistente;
  3. salvare in sessione almeno access_token, refresh_token, sub, local_user_id e dati profilo minimi;
  4. considerare l'utente autenticato sul sito client.

10. Durata token

Configurazione attuale:

  1. access token: 15 minuti;
  2. refresh token: 30 giorni;
  3. authorization code: 10 minuti, monouso.

11. Sicurezza implementata

Il sistema arcana-sso-v2 applica queste regole:

  1. niente password o hash password nelle API;
  2. niente login via query string con password;
  3. PKCE obbligatorio;
  4. redirect URI in whitelist;
  5. authorization code monouso;
  6. JWT firmato HS256;
  7. refresh token revocabile;
  8. refresh token rotation;
  9. logout front-channel sul dominio AD;
  10. separazione tra identita' globale AD e ID locali client.

12. Errori comuni di integrazione

12.1 invalid_redirect_uri

Causa tipica: la redirect_uri passata non e' identica a quella registrata nel client.

Verifica:

  1. schema http/https;
  2. host;
  3. porta;
  4. path esatto.

12.2 invalid_client

Causa tipica: client_id o client_secret errati.

12.3 invalid_grant

Causa tipica:

  1. code gia' usato;
  2. code scaduto;
  3. redirect_uri diversa da quella usata in /authorize;
  4. code_verifier non coerente con il code_challenge.

12.4 invalid_token

Causa tipica:

  1. access token scaduto;
  2. token malformato;
  3. token firmato male o audience non valida.

12.5 HTTP status 0 nel client locale

Causa tipica:

  1. certificato TLS self-signed di Local non accettato da cURL;
  2. problema DNS locale;
  3. endpoint non raggiungibile.

13. Flusso completo consigliato per i team esterni

  1. registrare il client su Arcana Domine;
  2. ottenere client_id e client_secret;
  3. registrare la redirect_uri esatta;
  4. generare state e PKCE per ogni login;
  5. chiamare /authorize dal browser;
  6. ricevere code e state nella callback;
  7. validare state;
  8. chiamare /token server-to-server;
  9. chiamare /userinfo;
  10. fare mapping sub -> local_user_id;
  11. creare sessione locale;
  12. al logout, revocare il refresh token;
  13. opzionalmente eseguire il logout front-channel su AD.

14. Esempio minimo lato client

1. Genera state
2. Genera code_verifier
3. Deriva code_challenge
4. Redirect browser a /authorize
5. Ricevi code e state in callback
6. Verifica state
7. POST /token
8. GET /userinfo
9. Login locale utente
10. POST /revoke al logout
11. Redirect a /?ad_sso_logout=1... per logout globale AD

15. Note finali per i sottodomini Arcana Domine

Per i sottodomini con database separati, questo sistema e' preferibile al legacy perche':

  1. evita di condividere direttamente l'ID numerico locale tra DB diversi;
  2. introduce un'identita' globale unica AD (sub);
  3. permette a ogni sito di mantenere il proprio utente locale;
  4. consente di usare siti WordPress o custom PHP con la stessa logica.

Regola fondamentale:

  1. non usare mai l'ID numerico locale di AD come ID locale del sito client;
  2. usare sempre sub o arcanadomine_id come chiave federata;
  3. risolvere poi il corrispondente local_user_id nel database del client.
